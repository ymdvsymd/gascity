## 問題

Gas City の現在の pack コマンドモデルは機能していますが、Pack/City v.next で求める import 指向の構造ではなく、依然として V1 構成モデルに形作られています。

今日、pack は `pack.toml` に `[[commands]]` を宣言でき、`gc` CLI はそれらのコマンドを解決された pack ディレクトリから発見し、以下のように登録します。

```text
gc <pack-name> <command-name>
```

これにより 8 つの問題が生じます。

1. **コマンド名前空間が pack 定義から来る、import バインディングからではない。** V2 では、pack は名前付き import を介して city に入ることになっています。ローカルバインディングが永続的なアイデンティティです。コマンドは依然として `[pack].name` をキーとしています。

2. **エイリアスがコマンドに伝わらない。** city が `gastown` を `gs` としてインポートしても、現在のモデルでは自然に `gc gs ...` を公開する方法がありません。コマンド表面はユーザーが選んだ構成名を無視します。

3. **フラット化が CLI を通してリークする。** V1 構成は pack をディレクトリに展開し、結果の pack ディレクトリからコマンドを発見します。これは実装に形作られたものであり、モデルに形作られたものではありません。

4. **衝突処理が不十分に仕様化されている。** 現在の実装は、pack 名がコア `gc` コマンドをシャドウすることに対してのみ保護します。インポートされたエイリアス、import 全体での重複コマンド名、または同じ pack を複数のバインディングで繰り返しインポートすることに対する正しいルールはまだ定義されていません。

5. **コマンド構造が pack モデルの他の部分よりも未熟。** エージェント、formula、プロンプト、import には今や明示的な設計の方向があります。コマンドは依然として、アイデンティティ、推移性、エクスポート、スコープのような基本的な質問に対する明確な答えを必要としています。

6. **pack は `gc` 表面領域への貢献方法に対する制御が少なすぎる。** 今日、コマンド定義はほとんど CLI 公開を意味します。pack がすべてを同じ方法でエクスポートすることなくコマンドを定義できるモデルが必要です。

7. **現在のモデルは実装の単位と CLI 公開の単位を混同している。** 単一のサブシステムは、複数のユーザー向けコマンドツリーに貢献しながら、名前付きセッション、プロバイダ、プロンプト、overlay、アセット、ヘルパーを共有することを望む場合があります。

8. **doctor とコマンドが構造的に並列であるにもかかわらず別々に設計されている。** 両方ともメタデータとアセットを持つ pack 提供のオペレーショナルエントリーポイントですが、それらに対するまとまりのあるストーリーはまだありません。

## 現在の状態

このセクションでは、再設計のためのしっかりした参照点を持つよう、現在の挙動を記述します。

### 定義

V1 pack は `pack.toml` でコマンドを直接宣言できます。

```toml
[[commands]]
name = "status"
description = "Show pack status"
long_description = "commands/status-help.txt"
script = "commands/status.sh"
```

実装は現在、コマンドを最小限のメタデータを持つスクリプトバックリーフとして扱います。

- `name`
- `description`
- `long_description`
- `script`

### 発見と登録

CLI 起動時、`gc` は以下を行います。

1. city config をロードする
2. すべての解決された pack ディレクトリを収集する
3. 各 `pack.toml` から `[[commands]]` を読む
4. エントリを pack の `[pack].name` でグループ化する
5. 各 pack のためのトップレベル名前空間コマンドを登録する

つまり、可視の CLI 形態は以下です。

```text
gc gastown status
gc dolt logs
gc dolt sql
```

### 起動モデル

各コマンドは現在、提供する pack ディレクトリからスクリプトを以下で実行します。

- argv のパススルー
- stdio のパススルー
- 小さな city と pack 環境変数のセット

これは意図的にシンプルで、pack 固有のオペレーショナルツールのエスケープハッチとしてうまく機能してきました。

### 現在の限界

現在のモデルはまだ以下に答えていません。

- コマンドが pack 定義に属するのか import バインディングに属するのか
- コマンドが推移的に再エクスポートされるか
- エイリアスがコマンド名にどう影響すべきか
- 同じ pack が 2 度インポートされた場合に 2 つのコマンド名前空間を生成するか
- city ローカルコマンドとインポートされたコマンドが同じネーミングルールに従うべきか
- rig import が rig 固有のコマンドを公開できるか
- pack が `gc import ...` のようなトップレベルコマンドツリーに貢献できるか
- コマンドと doctor チェックがどのように構造を共有すべきか

## 設計原則

再設計は Pack/City v.next と同じ原則に従うべきです。

### 0. コマンドの挙動は V2 ではローダー形態であるべき。後付けではない

この再設計は、コマンドを後付けでボルトオンするのではなく、import とローダーの変更を吸収する必要があります。

つまり、

- ローダーは pack ロードの一部としてコマンドを発見すべき
- 構成済みコマンド表面は import グラフから派生すべき
- city pack コマンドとインポートされた pack コマンドは 1 つのモデルで扱われるべき
- CLI は構成済みモデルが公開するものを登録すべき。後で別の pack-dir ビューを再発見すべきではない

V2 ローダーが「この city は `gs` をインポートし、`maintenance` を再エクスポートしているがそのコマンドは再エクスポートしていない」と言う場合、コマンド表面はそのモデルに直接従うべきです。

### 1. コマンドは pack によって定義されるが、import を通じて公開される

pack は自身のコマンド定義を所有します。city または pack import は、それらのコマンドが構成済み CLI 表面でどのように可視になるかを決定します。

これが中心的なシフトです。

- V1 の考え方: 「`gastown` pack が `gc gastown ...` を作る」
- V2 の考え方: 「`gastown` import バインディングがコマンド名前空間を公開する」

### 2. import バインディングが公開名前空間

pack が以下のようにインポートされた場合:

```toml
[imports.gs]
source = "./packs/gastown"
```

そのコマンドは pack の内部 `[pack].name` ではなく `gs` の下に公開されるべきです。

ユーザー向けの CLI 表面は city が構成された方法を反映すべきです。

### 3. コマンド構造はモデル形態であるべき。キャッシュ形態ではない

ローダーと CLI は依然として pack をディレクトリにマテリアライズし、それらのディレクトリからスクリプトを実行することができますが、それは実装詳細であるべきです。概念モデルは、import、バインディング、エクスポートされたコマンド表面の観点から表現されるべきです。

### 4. コマンドは他のインポートされたコンテンツと同じ閉包ルールに従うべき

V2 import がデフォルトで閉じている場合、コマンドもデフォルトで閉じているべきです。pack の内部 import は、そのエクスポートが明示的でない限り、消費者の CLI 表面をサイレントに拡大するべきではありません。

### 5. シンプルな起動は依然として機能

現在のスクリプトバック実行モデルは便利であり、置き換える強い理由がない限り維持すべきです。コアモデルを破棄することなく、後でより豊かなメタデータを追加できます。

重要なニュアンスは、「スクリプトバック」が特別なトップレベル `scripts/` 表面を意味しないことです。V2 では、コマンドエントリーポイントは `commands/` 配下に存在でき、任意のヘルパー実行可能ファイルは不透明な `assets/` ディレクトリ配下に存在し、相対パスで参照できます。

### 6. すべての pack はコマンドを定義できる、ルート city pack を含む

city pack は依然として pack です。構成のルートだからといって、コマンドのために特別な 1 回限りのメカニズムを必要とすべきではありません。

モデルは以下をサポートするべきです。

- インポートされた pack によって定義されるコマンド
- 明示的に再エクスポートされた場合の推移的なインポート pack によって定義されるコマンド
- ルート city pack 自身によって定義されるコマンド

これらのケースの違いは、まったく別のメカニズムではなく、公開と名前空間のものであるべきです。

### 7. pack は `gc` 表面領域への貢献方法に対する明示的な制御が必要

コマンドの定義とコマンドの公開は関連していても、同一のコンセプトであるべきではありません。

おそらく、pack が以下の組み合わせを言える方法が必要です。

- このコマンドはローカル/内部用途で存在する
- このコマンドは pack が直接インポートされたときに通常公開される
- このコマンドは明示的に再エクスポートされない限り隠されたままであるべき
- この pack は特定の名前空間または公開スタイルを好む（importer の制御に従って）

importer は依然として最終構成を所有しますが、pack は意図を表現できるべきです。

### 8. コマンドの配置と実装は分離可能であるべき

1 つの pack は、以下を共有することを望むなら、合理的に複数の外向き機能を実装する場合があります。

- 名前付きセッション
- プロバイダ
- プロンプト
- overlay
- ローカルアセット
- ヘルパーエージェント

つまり、「1 pack = 1 コマンドルート」を厳格なルールとして強制するべきではありません。

pack の実装境界と CLI 配置境界は関連していますが、同じコンセプトではありません。

### 9. コマンドと doctor チェックは 1 つの構造パターンを使うべき

コマンドと doctor チェックは兄弟表面のように見えます。

- 名前付きオペレーショナルエントリ
- エントリーポイント
- メタデータ
- 任意のヘルプ
- ローカルアセット

同じ種類のものに 2 つの無関係な規約を発明しないように、それらは一緒に設計されるべきです。

## 提案される方向

このセクションは新しい設計の現在の状態を捉えており、最終ロックされた仕様ではありません。

### ローダー統合

V2 では、コマンドは city モデルの残りを構築するのと同じ構成パスの一部としてロードされるべきです。

概念的には:

1. ルート city pack がロードされる
2. その直接 import が解決されロードされる
3. 各 pack はコマンド定義に加えてコマンド公開メタデータを貢献する
4. ローダーは import グラフから構成済みコマンド表面を計算する
5. CLI はその構成済みコマンド表面からコマンドを登録する

これは、以下の現在の分割を置き換えます。

- ローダーがランタイム挙動のために pack を解決する
- CLI が別途 pack ディレクトリをスキャンしコマンド名前空間を再構築する

その分離は V1 では受け入れられますが、V2 では誤った抽象化になります。

### コマンドアイデンティティ

コマンドは 2 つのアイデンティティを持ちます。

- **定義アイデンティティ**
  - pack によって宣言されたコマンド
- **公開アイデンティティ**
  - import バインディングを通じて利用可能になったコマンド

pack は以下を宣言します。

```text
status
logs
sql
```

import する city はそれらを以下のように公開します。

```text
gc <binding> <command>
```

例:

```text
gc gastown status
gc gs status
gc dolt sql
```

ここで `gastown`、`gs`、`dolt` は import バインディングであり、内在的な pack 名ではありません。

ルート city pack もコマンドアイデンティティを持ちますが、その公開ルールは別です。なぜなら、それは import バインディングではなく CLI 表面のルートだからです。

### 公開モード

現在好まれる方向は、コマンドを以下の 2 つの広い方法のいずれかで配置できることです。

1. **バインディングスコープ**
   - import バインディングの下に公開
   - 例: `gc gs status`
2. **ルート拡張**
   - 承認されたトップレベルコマンドツリーの下に公開
   - 例: `gc import add`
   - 例: `gc import upgrade`

これにより、デフォルトでバインディングモデルが利用可能のままで、pack 境界に不器用に縛られない共有製品表面のための余地が残ります。

### 公開ルール

現在好まれる方向は以下です。

1. **コマンドは直接 import を通じて公開される。**
2. **コマンドはデフォルトで推移的に公開されない。** これは一般的な import モデルから逸脱します。エージェントはデフォルトで推移的だからです。理由: CLI 表面の公開は、エージェント構成よりも衝突と発見可能性のリスクが高いです。エージェントは qualified 名前空間に構成されますが、コマンドはユーザー向けの動詞空間を競い合います。
3. **コマンドの再エクスポートは、サポートされる場合、明示的であり、コマンド専用のメカニズムを発明するのではなく、pack 再エクスポートのセマンティクスに従うべき。**

これにより、コマンドの挙動を広範な import モデルに合わせながら、推移的公開について意図的により保守的に保ちます。

バインディングスコープのコマンドでは、バインディングが名前空間です。ルート拡張コマンドでは、トップレベルツリーが名前空間です。

### pack の貢献と公開制御

以下の区別が必要です。

- **コマンド定義**
  - 「この pack はコマンドを実装する」
- **デフォルト公開**
  - 「このコマンドは pack が直接インポートされたときに通常公開される」
- **再エクスポートの挙動**
  - 「このコマンドは親 pack を通じて再公開できる/できない」

正確なスキーマはまだオープンですが、概念的なニーズは明確です。pack は、「すべての宣言されたコマンドが公開になる」を超えて、`gc` への貢献方法に対するより多くの制御を持つべきです。

可能な制御には以下が含まれます。

- pack レベルのコマンドエクスポートポリシー
- コマンドごとの可視性またはエクスポートフラグ
- importer 側のフィルタリングまたは再マッピング

これを早すぎる段階で過剰設計しないよう注意すべきですが、設計はそのための場所を必要とします。

### 拡張ルート

pack がトップレベルコマンドツリー下に貢献することを許可される場合、それは組み込みコマンドへの任意のアタッチメントではなく、明示的な拡張ポイントを通じて行われるべきです。

ありそうなカテゴリは:

1. **所有ルート**
   - デフォルトで pack 拡張に開かれていない組み込みコマンドツリー
2. **拡張ルート**
   - pack の貢献を明示的に意図したトップレベルツリー
3. **バインディングルート**
   - import バインディングをルートとするコマンドツリー

これにより、任意のインポートされた pack が任意の組み込みコマンド下に静かにアタッチできる世界を避けます。

### 繰り返しの import とエイリアス

同じ pack が異なるバインディングで複数回インポートされる場合、各バインディングは独自のコマンド名前空間を公開できるべきです。

例:

```toml
[imports.prod]
source = "github.com/example/deploy-pack"

[imports.staging]
source = "github.com/example/deploy-pack"
```

そして両方が存在できます:

```text
gc prod status
gc staging status
```

これは機能であり、バグではありません。コマンド名前空間はバインディングに属します。

### city ローカルコマンド

city pack 自身がコマンドを定義できるべきです。正確な公開形態はまだオープンですが、ありそうなオプションは:

1. トップレベル `gc <command>` として公開
2. 明示的なローカル名前空間下に公開
3. ルート city pack のために予約された区別される名前空間

トップレベルコマンド公開は、インポートされた名前空間と異なる衝突と発見可能性のトレードオフを持つため、これは意図的な選択を必要とします。

もはやオープンでないのは、city pack がそもそもコマンドを定義できるかどうかです: できるべきです。

### マニフェストとオンディスクの形

現在好まれる方向は、doctor と並列に、任意のマニフェストメタデータを持つディレクトリ形のコマンドツリーです。

```text
commands/
├── status/
│   ├── run.sh
│   └── help.md
└── repo/
    └── sync/
        ├── run.sh
        └── help.md
```

主要なプロパティ:

- ネストされたディレクトリはネストされたコマンドワードを意味する
- `run.sh` はデフォルトの well-known エントリーポイント
- `help.md` は存在する場合のデフォルトの well-known ヘルプファイル
- エントリローカルなスクリプトとヘルプはエントリーポイントの隣に存在
- コマンドローカルなアセットは同じディレクトリに存在できる
- 最もシンプルなケースは TOML なしで動作すべき
- `command.toml` はデフォルトのディレクトリベースのマッピングが不十分なときのみ現れるべき

これにより、ファイルシステム形態を CLI 形態に合わせながら、各コマンドリーフのローカルアセットスコープを許可します。

意図された進行は:

1. **ゼロコンフィグコマンド**
   - `commands/status/run.sh` は `status` になる
   - `commands/repo/sync/run.sh` は `repo sync` になる
2. **マニフェスト支援コマンド**
   - メタデータまたは明示的なオーバーライドが必要なときのみ `command.toml` が現れる

### マニフェストフィールドの方向

現在好まれる方向は、ファイルシステムがデフォルトのコマンドワードを提供し、マニフェストはコマンド配置の主要なソースではなくエスケープハッチであることです。

つまり:

- ネストされたディレクトリがデフォルトのユーザー向けコマンドワードを提供する
- 実行可能エントリーポイントは、デフォルトの `run.sh` が存在する場合、通常は明示的な TOML フィールドを必要とすべきではない
- マニフェスト自体はシンプルなケースではオプションであるべき
- 後でオーバーライドフィールドが必要になった場合、それは意図的に導入されるべきであり、ディレクトリレイアウトでクリーンに表現できないケースのみのため

これにより、共通ケースをツリーから明白に保ちながら、デフォルトでは不十分なときのために明示的なメタデータの余地を残します。

### import コマンドの例

import コマンドは、以下のいずれかとしてモデル化できるため、有用なストレステストになります。

- 2 つの異なる pack
- 同じトップレベルコマンドツリーに貢献する 1 つの共有実装 pack

#### ケース 1: 別々の pack

```toml
[imports.import_add]
source = "https://github.com/gastownhall/gc-import-add"

[imports.import_upgrade]
source = "https://github.com/gastownhall/gc-import-upgrade"
```

add pack のコマンド:

```text
commands/import/add/run.sh
```

upgrade pack のコマンド:

```text
commands/import/upgrade/run.sh
```

結果:

```text
gc import add
gc import upgrade
```

これは、ユーザー向け機能の分割と pack の分割が同じであるときの最もクリーンなケースです。

ネストされたコマンドはディレクトリ形でも自然に動作するべきです。例:

```text
commands/repo/sync/run.sh
```

は以下になります。

```text
gc <binding> repo sync
```

`command.toml` はデフォルトのマッピングが不十分なときのみ追加されます。

#### ケース 2: 1 つの共有実装 pack

```toml
[imports.packman]
source = "https://github.com/gastownhall/gc-packman"
```

この pack がそのバインディング下にのみ貢献する場合、結果は以下になります。

```text
gc packman import add
gc packman import upgrade
```

この pack が拡張ルートに貢献することが許可される場合、代わりに以下の下に到達するエントリを定義できます。

```text
gc import add
gc import upgrade
```

これがコマンド配置と pack 実装が分離可能であるべき理由です。

### 実行モデル

現在のデフォルトの仮定は、コマンドはスクリプトバックのままであることです。

```text
commands/status/
└── run.sh
```

ローダー/CLI は以下を行うべきです。

- 存在する場合、`run.sh` をデフォルトのエントリーポイントとして扱う
- デフォルトが不十分なときのみ明示的なオーバーライドを許可する
- マニフェストが存在する場合、コマンドローカル参照をマニフェストファイルからの相対で解決する
- それ以外はコマンドエントリディレクトリからの相対で解決する
- pack と city コンテキストで解決されたエントリーポイントを実行する

これにより、pack コマンドを軽量でオペレーショナルに有用に保ちながら、後でより豊かなメタデータの余地を残します。

エントリローカルマニフェストには、より具体的な好まれるルールがあります。

- デフォルトの `run.sh` はコマンドリーフディレクトリから解決される
- 明示的な `run = "..."` オーバーライドは、サポートされる場合、コマンドリーフディレクトリからの相対でも解決される
- その他のコマンドローカルなアセット参照はコマンドリーフディレクトリからの相対で解決される
- 正規化されたパスが同じ pack ルート内にとどまる限り、`..` は許可される

マニフェストなしの場合、同等のデフォルトルールは:

- コマンドリーフディレクトリがベース
- `commands/` からそのリーフへの相対パスがデフォルトのコマンドワードを提供
- `run.sh` がデフォルトのエントリーポイント

それは pack 境界を厳格に保ちながらローカリティを保持します。

### ディレクトリ規約

この提案は、コマンド定義が [doc-directory-conventions.md](doc-directory-conventions.md) で記述された V2 pack ディレクトリ規約にスナップすると仮定します。

特に:

- コマンドは `commands/` 配下に属する
- doctor チェックはコマンドと並んだ兄弟オペレーショナル表面として一緒に設計されるべき
- コマンドアセットは無関係なディレクトリに散らばらず、コマンドとともに存在するべき
- city pack とインポートされた pack は同じオンディスク規約に従うべき
- 任意のヘルパーファイルは特別な pack 全体の `scripts/` 規約を要求せず `assets/` 配下に存在するべき

コマンドモデルは特別な V1 専用のディレクトリレイアウトを要求するべきではありません。

## オープンな質問

これらはこのドラフト時点での主な未解決の設計上の質問です。

### 1. コマンド公開はどの正確な構文を使うべきか?

現在好まれる形態は:

```text
gc <binding> <command> [args...]
```

オープンな代替案:

- `gc command <binding> <command>`
- `gc <binding>:<command>`
- `gc run <binding>.<command>`

専用の動詞下に集中させる強い理由を発見しない限り、デフォルトはおそらく今日の `gc <namespace> <leaf>` 形態に近いままにすべきです。

### 2. city pack コマンドはどのように公開されるべきか?

オプションは以下を含みます。

1. トップレベルコマンド
2. `gc city <command>` のような予約された名前空間
3. ルート pack 名から派生した名前空間

これは主に人間工学対衝突安全性の問題です。

### 2a. pack は公開に対してどれだけの制御を得るか?

主要なノブがどこに存在するかをまだ決定する必要があります。

1. importer 側のみ
2. 主に pack 側で importer オーバーライド
3. pack デフォルトと importer ポリシーのミックス

私の現在の傾向は、pack がデフォルトを表現し、importer が最終公開を所有することです。

### 2b. どのトップレベルコマンドツリーが拡張ルートか?

pack がトップレベルツリー下に貢献する可能性がある場合、どのルートが拡張に開かれ、どれが保護されるかを決定する必要があります。

現在の傾向:

- すべての組み込みルートが暗黙的に拡張可能であるべきではない
- 拡張ルートは明示的であるべき
- バインディングスコープの公開は常に利用可能のままであるべき

### 3. コマンド再エクスポートはフラット化すべきか provenance を保持すべきか?

pack `A` が pack `B` を再エクスポートし、`B` が `status` コマンドを提供する場合、ユーザーは以下を見るべきでしょうか:

```text
gc a status
```

それともネストされた provenance が何らかの形で可視のまま残るべきでしょうか?

これはエージェントと formula の再エクスポートネーミングと同じファミリーの質問であり、おそらく一貫して答えられるべきです。

### 4. rig レベルコマンドはスコープに含まれるか?

city import がコマンドを公開できる場合、rig import もコマンドを公開すべきでしょうか? もしそうなら、アドレス形式は何でしょうか?

候補:

- `gc <rig> <binding> <command>`
- `gc rig <rig> <binding> <command>`
- コマンドは city スコープのみのまま

私の現在の傾向は、まずコマンドを city スコープに保ち、実際のユースケースが要求した場合のみ rig スコープのコマンド公開を追加することです。

### 5. コマンドはスクリプト実行を超えてどのメタデータをサポートすべきか?

可能な将来のフィールド:

- 引数スキーマ
- マシン可読ヘルプ
- プロバイダ/ランタイム要件
- ワーキングディレクトリポリシー
- 出力モード
- インタラクティブ vs 非インタラクティブ宣言

今のところの質問は、この再設計の V1 でスキーマを意図的に小さく保ち、後で構造を追加するべきかどうかです。

### 5a. コマンドは `pack.toml` の `[[commands]]` のままか、規約ベースになるか?

より広範な V2 の方向は、可能な場合、明示的な TOML 宣言よりもファイルシステム規約を優先します。

オープンなオプション:

1. `[[commands]]` を `pack.toml` に保つ
2. `commands/` 配下のエントリごとのディレクトリでコマンドを定義する
3. 発見のために規約を使い、加えて任意の小さなエントリごとのマニフェスト

この質問は今やディレクトリ規約ドキュメントに依存し、エージェント、formula、スクリプト、doctor チェックと一貫して答えられるべきです。

### 5b. コマンドと doctor チェックは 1 つの基本形を共有すべきか?

現在の傾向: イエス。

それらは同じ構造概念の 2 つのバリアントのように見えます。

- 名前付きオペレーショナル表面
- エントリーポイント
- メタデータ
- ヘルプテキスト
- ローカルアセット

それらが 1 つの形を共有する場合、`commands/` 用と `doctor/` 用に完全に異なる規約を作ることを避けるべきです。

### 5c. マニフェストフィールド名は何であるべきか?

特に:

- ユーザー向けコマンドワードのオーバーライドフィールドが必要か?
- 配置モードを何と呼ぶか?
- 拡張ルートターゲティングを何と呼ぶか?

現在の傾向:

- 共通ケースではコマンドワードフィールドを要求しない
- ディレクトリ形がデフォルトのコマンドワードを提供させる
- 可能な場合、`run.sh` や `help.md` のような明白なローカルファイルにファイル名規約を使う
- ディレクトリ形がうまく表現できない実際の配置ニーズが現れた場合のみ明示的なオーバーライドフィールドを追加する

### 6. 衝突はどのように扱われるべきか?

以下のための明示的なルールが必要です。

- インポートされたバインディングがコア `gc` コマンドと衝突
- city ローカルコマンドがコアコマンドと衝突
- 単一の名前空間内のコマンド名衝突
- 再エクスポートによって導入される衝突

ありそうな方向:

- コアコマンドは常に勝つ
- 名前空間衝突はロード時エラー
- 名前空間内のリーフ衝突は、構成モデルによって 1 つのソースが明示的にシャドウされない限りエラー

それは依然として慎重に仕様化される必要があります。

## 非ゴール

このドキュメントは現在以下を解決しようとしていません。

- パッケージ発見またはカタログ表面
- リモートフェッチとロックファイル挙動の詳細
- 完全な構造化コマンド引数 DSL
- プロバイダ固有のコマンド実行の抽象化
- オペレーショナルタスクのための通常のシェルスクリプトの置き換え

それらは後でコマンド設計に接続するかもしれませんが、このノートの焦点ではありません。

## 作業中ドラフトのまとめ

現在の設計の方向は:

1. pack はコマンドを定義する
2. ローダーは import グラフを通じてコマンドを構成する
3. import はコマンドを公開する
4. import バインディング、pack 名ではなく、コマンド名前空間を定義する
5. ルート city pack を含むすべての pack はコマンドを定義できる
6. pack は `gc` 表面領域への貢献方法に対する明示的な制御が必要
7. コマンド配置と pack 実装は分離可能であるべき
8. 一部のトップレベルコマンドツリーは明示的な拡張ルートになる可能性がある
9. コマンドと doctor チェックは兄弟オペレーショナル表面として設計されるべき
10. コマンドは import モデルの残りとともにデフォルトで閉じている
11. スクリプトバック実行はデフォルトの開始点のままで、`run.sh` と `help.md` がありそうなデフォルトのローカル規約となり、特別な pack 全体の `scripts/` ディレクトリは不要

主な未解決領域は、コマンド公開構文、city pack 公開、拡張ルートポリシー、pack ごとの公開制御、マニフェストフィールドのネーミング、convention vs TOML 宣言、共有 command/doctor 形態、再エクスポート挙動、rig スコープ、衝突ポリシーです。
