---
title: Dolt Bloat Recovery
description: Dolt の noms ディレクトリが肥大化した Gas City の beads ストアを復旧する。
---

## 概要

Gas City は beads を管理対象の Dolt サーバに保存します。Dolt はすべての書き込みを `.beads/dolt/<database>/.dolt/noms/` 配下のイミュータブルなチャンクとして記録します。チャンクはガベージコレクションによってのみ回収されます。Dolt の自動 GC は前回の GC から ~125 MB 程度の新しいチャンクが蓄積されたときに発火するため、エージェントの嵐で一度肥大化したのちに静かになったデータベースは自動 GC が走らず、ピークサイズで止まったままになります。

このランブックは、オペレータがそのようなデータベースを復旧する手順を案内します。書き込み停止、安全バックアップの取得、アーカイブ圧縮を伴うフル GC の実行、結果の検証までをカバーします。

## 症状

1. `gc doctor` が `dolt-noms-size` を **Warning** または **Error** として報告する。
2. `du -sh <cityPath>/.beads/dolt/<database>/.dolt/` が 20 GB を超える。
3. `bd` の書き込みが普段より遅く感じる、または `bd ready` に明らかな時間がかかる。
4. エージェントが Dolt の接続またはタイムアウトエラーで失敗する（特に起動直後）。

## 前提条件

- **すべてのエージェントを停止する。** 影響を受ける city で `gc stop` を実行し、Dolt に書き込む session が無いようにします。
- **外部ライターが接続されていないことを確認する。** 管理対象ポートに対して `dolt sql` シェルを開いている場合は終了します。
- **空きディスク容量を確保する。** Dolt の GC はチャンクを新しいストアに書き直してからスワップします。同じファイルシステム上に少なくとも **現在の `.dolt/` サイズの 2 倍** の空きを用意してください。
- **Dolt 1.86.1 以降。** これは Gas City の管理対象 Dolt ツールが強制する下限と一致し、pack が使うリスナー/コンフィグつまみと最新の自動 GC 挙動が利用可能であることを保証します。`dolt version` で確認してください。お使いのバイナリが `--archive-level=1` を拒否する場合（最新リリースではほぼ無い）はフラグを外して素の `dolt gc` を実行してください。アーカイブ圧縮は 1.75+ ではデフォルトで有効なので、フラグは最適化であり必須ではありません。

## 復旧手順

```bash
# 1. supervisor を停止する（同時にすべてのエージェントと管理対象 Dolt サーバも停止）。
gc stop <cityPath>

# 2. ストアに触れる前に安全バックアップを取得する。
cd <cityPath>/.beads/dolt/<database>
cp -a .dolt .dolt.bak-$(date +%Y%m%d-%H%M%S)

# 3. アーカイブ圧縮を伴うフル GC を実行する。これが実際に容量を回収するステップ。
#    120 GB のストアでは数十分かかることを想定。
dolt gc --archive-level=1

# 4. city を再起動して検証する。
cd <cityPath>
gc start
gc doctor          # dolt-noms-size は OK になっているはず
du -sh .beads/dolt/<database>/.dolt
```

`gc doctor` が `dolt-noms-size` を綺麗に報告し、エージェントが問題なく復旧した場合、復旧は完了です。新しいストアに自信が持てたら、`.dolt.bak-*` ディレクトリは適宜削除してかまいません。

## 期待される結果

DoltHub のアーカイブフォーマットは通常、通常の GC に加えて ~30% の圧縮率を実現します（[DoltHub blog, archive storage](https://www.dolthub.com/blog/)）。エージェントのチャーンによる孤児チャンクの回収と組み合わせることで、120 GB の GC 前ストアは通常 **5 GB から 20 GB の間** に縮小します。GC 前のサイズのうちライブデータと孤児チャンクの比率に依存します。

GC は完了したがサイズがほとんど変わらない場合、チャンクのほぼ全てが生きている（回収できるガベージがない）ことを意味します。下記の **エスカレーションのタイミング** を参照してください。

## 予防

- **Dolt を 1.86.1 以降に保つ。** これは Gas City の管理対象 Dolt の下限と一致し、新しいリリースには改善された自動 GC ヒューリスティクスとデフォルトのアーカイブ圧縮が含まれます。
- **dolt pack の `dolt-gc-nudge` order を継続的に動かす。** これは dolt pack に組み込まれて出荷され、デフォルトで `CALL DOLT_GC()` を 1 時間ごとに無条件で実行します。Gas City の管理対象 Dolt 起動パスは現在 `DOLT_GC_SCHEDULER=NONE` を強制し、[dolthub/dolt#10944](https://github.com/dolthub/dolt/issues/10944) に影響を受ける多コアホストでも Dolt の設定された自動 GC 挙動を復元します。1 時間おきの nudge は bd ワークロードに対するベルト＆サスペンダーのバックストップとして、また閾値駆動の自動 GC が一定期間何もすることがない場合の無条件の復旧経路として依然として価値があります。GC は冪等であり、回収するものがないときはほぼ無料なので、毎時実行しても安価です。特定の city で無効化するには、city の `[orders] skip = [...]` リスト（または rig レベルの `[[order.override]]`）に `dolt-gc-nudge` を追加してください。小さなデータベースで GC をスキップするには、city の環境で `GC_DOLT_GC_THRESHOLD_BYTES` を正のバイト数に設定してください（デフォルト: 0 — 無条件で実行）。
- **`orders.max_timeout` を設定する場合は注意する。** nudge order は大きなストアに対する直列化された `CALL DOLT_GC()` 実行に対応するため 24 時間のタイムアウトを要求します。city レベルの `orders.max_timeout` が 24 時間未満だと nudge が打ち切られ、進行中の GC を kill する可能性があります。大きなデータベースでの自動復旧を望むなら上限を上げるか未設定のままにしてください。
- **`gc doctor` を定期的に実行する。** 日次の cron や CI ジョブで十分です。`dolt-noms-size` チェックは、ユーザが気づくよりかなり早く警告を出します。
- **Gas City の外部からの長寿命な `dolt sql` セッションを避ける。** 外部クライアントは GC をブロックし得るオープントランザクションを保持します。

## エスカレーションのタイミング

復旧 GC を実行してもストアが ~10% 未満しか縮小せず、`gc doctor` が依然として `dolt-noms-size` をフラグする場合:

1. 残っているチャンクはおそらくすべて生きている — データベースには本当にこれだけの履歴があるということです。Dolt の履歴をスカッシュすることは現状サポートされたセルフサービス操作ではないので、代わりにエスカレーションしてください。
2. `bd` イシューを以下と共に提出してください:
   - `dolt version` の出力
   - `.dolt/` ディレクトリの `du -sh`
   - `dolt log --oneline | wc -l`
   - 最も忙しい日からの `dolt log --stat` のサンプル

`gc doctor --verbose` の出力も添付してください。イシューがオープンの間は `.dolt.bak-*` ディレクトリを削除しないでください。
