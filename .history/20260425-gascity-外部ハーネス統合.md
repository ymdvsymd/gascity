# 2026-04-25 Gas City 調査: 外部ハーネス統合

> リポジトリ: `/Users/to.watanabe/oss/gascity` (Gas City — Go 製 orchestration-builder SDK)

---

## 調査の問い

ユーザーが Claude Code / Codex 等の AI エージェント CLI を併用しており、それぞれに対してルール・hook 等の「外部ハーネス」を個別に設定するのが面倒。

**Q: Gas City はこの問題を解決するか？ つまり、Claude Code / Codex などの外部ハーネスを抽象化して、統一的に設定できる仕組みを提供するか？**

---

## 結論

**部分的に YES。ただし「Gas City の orchestration を採用するならば」という条件付き。**

| 観点 | 回答 |
|------|------|
| 8 プロバイダ用 hook ファイルを 1 設定から自動生成する | **YES** |
| 役割（prompt template）を 1 つ書けば全 CLI で動く | **YES** |
| 意味的オプション（permission_mode, model, effort 等）を CLI 別フラグに自動射影する | **YES** |
| 複数 CLI を 1 つの city で混在運用できる | **YES** |
| ユーザーが任意の **カスタムルール**（例: "rm -rf 禁止"）を 1 箇所に書いて全 CLI に展開する DSL | **NO** |
| 配置される hook の中身は Gas City の orchestration 専用（`gc prime --hook` 等） | hard-coded |

## 抽象化の中身（YES の側）

### 1. プロバイダ・カタログ

`internal/worker/builtin/profiles.go:77-80`:
```go
var builtinProviderOrder = []string{
    "claude", "codex", "gemini", "cursor", "copilot",
    "amp", "opencode", "auggie", "pi", "omp",
}
```

各プロバイダに `BuiltinProviderSpec` (`profiles.go:30-58`) が定義され、
`Command`, `PromptMode`, `SupportsHooks`, `InstructionsFile`, `ResumeFlag/Style`,
`PermissionModes`, `OptionsSchema` を持つ。

意味的オプションの CLI フラグ射影例:

- Claude: `permission_mode = "unrestricted"` → `--dangerously-skip-permissions`（`profiles.go:103-108`）
- Codex: 同じ `permission_mode = "unrestricted"` → `--dangerously-bypass-approvals-and-sandbox`（`profiles.go:165-169`）

→ **同じセマンティック選択が CLI 別フラグに翻訳される**。

### 2. hook 自動インストール

`internal/hooks/hooks.go:113-130` の `InstallWithResolver`:
```go
switch family {
case "claude":
    err = installClaude(fs, cityDir)
case "codex", "gemini", "opencode", "copilot", "cursor", "pi", "omp":
    err = installOverlayManaged(fs, workDir, family)
}
```

各 CLI ネイティブな配置に展開される実体ファイル（`internal/bootstrap/packs/core/overlay/per-provider/`）:

| プロバイダ | 配置先 |
|-----------|--------|
| codex | `.codex/hooks.json` |
| cursor | `.cursor/hooks.json` |
| gemini | `.gemini/settings.json` |
| copilot | `.github/copilot-instructions.md`, `.github/hooks/gascity.json` |
| opencode | `.opencode/plugins/gascity.js` |
| pi | `.pi/extensions/gc-hooks.js` |
| omp | `.omp/hooks/gc-hook.ts` |
| claude | `<city>/.gc/settings.json`, `<city>/hooks/claude.json` |

### 3. ライフサイクル・イベントの正規化

CLI 間でイベント名表記が異なるが、Gas City は同じ `gc <subcommand>` を呼ばせて正規化:

| 概念 | Claude | Codex | Cursor |
|------|--------|-------|--------|
| 起動時 | `SessionStart` | `SessionStart` | `sessionStart` |
| 圧縮前 | `PreCompact` | – | `preCompact` |
| プロンプト送信前 | `UserPromptSubmit` | `UserPromptSubmit` | `beforeSubmitPrompt` |
| 停止 | `Stop` | `Stop` | `stop` |

呼ばれる Gas City コマンド:
`gc prime --hook` / `gc nudge drain --inject` / `gc mail check --inject` / `gc handoff "context cycle"` / `gc hook --inject`

### 4. プロンプトはプロバイダ非依存

`cmd/gc/prompt.go` の `renderPrompt()` が Go `text/template` でレンダリング。
`PromptContext`（`CityRoot`, `AgentName`, `WorkDir` 等）にプロバイダ識別子はない。

`examples/swarm/city.toml:14-17`:
```toml
[workspace]
name = "swarm"
provider = "claude"
includes = ["packs/swarm"]
```
ここで `provider = "codex"` に書き換えるだけで、同じ pack（同じ役割定義・同じプロンプト）が Codex 上で動く。

## NO の側（汎用ハーネス DSL は無い）

- 各 CLI に書き込まれる hook はすべて `gc <subcommand>` を呼ぶ glue で固定。
- ユーザーが「PreToolUse で `rm -rf` をブロック」のような任意ルールを 1 箇所に書いて全 CLI に展開する**汎用機構は存在しない**。
- これは設計思想（**ZFC = Zero Framework Cognition**）が意図的に拒否している。`AGENTS.md` で「No skills system」「No MCP/tool registration」「No decision logic in Go」を exclusion list に明記。
- ただし `installClaude` はマージ戦略を持ち（`hooks.go:160-167`）、ユーザー定義の `<city>/.claude/settings.json` を残しつつ Gas City の埋め込みデフォルトをマージするので、Claude については user カスタム hook と Gas City の glue は共存可能。

---

## ユーザーの "面倒" タイプ別の処方箋

| ユーザーの面倒 | Gas City で解けるか |
|----------------|---------------------|
| 複数エージェントを協調させたい、各 CLI の hook 配線が面倒 | **解ける**。`city.toml` + `pack.toml` + prompt template 1 セットで済む |
| 単独 CLI 利用のまま、共通ガードレール（例: `rm -rf` 禁止）を全 CLI に展開したい | **解けない**。汎用 hook DSL は無い |
| Claude / Codex 切替時にモデル・権限フラグの差を毎回調べるのが面倒 | **解ける**。`OptionsSchema` がフラグ差を吸収 |

## 次のアクション候補

- **試したい場合**: `examples/swarm/` を雛形にして `gc start examples/swarm/`（`AGENTS.md` の城ディレクトリモデル参照）。
- **ガードレール DSL 寄りのニーズなら**: Gas City の `installOverlayManaged` の発想を借りた最小ツールを自作する、または各 CLI の hook を雛形化してシェルスクリプトで配布する別解を検討。
- **混在運用の面倒さの正体をもっと掘りたい**: どの CLI のどの面倒（model 切替、効率切替、ガードレール、プロンプト共通化）が痛いかを切り分ければ、Gas City で解ける範囲・解けない範囲が明確になる。
