# 2026-04-25 Gas City 調査: 個別エージェントチャット機能

> リポジトリ: `/Users/to.watanabe/oss/gascity` (Gas City — Go 製 orchestration-builder SDK)

---

## 調査の問い

**Q: Gas City 自体に、各エージェント（例: 市長役のエージェント）と個別にチャットする機能（Claude Code のような対話 UI）はあるか？**

---

## 結論

**YES — ある。ただし主に CLI / tmux 経由。Web ブラウザ上のチャット UI は現状無し（観測のみ）。**

Gas City のチャット UX は **「Claude Code そのもののインタラクティブ画面 (tmux) を、役割ごとに永続化して切り替える」** という設計。役割ごとに別ペインで Claude Code が走り、`gc session attach mayor` でいつでも入って対話、デタッチで離脱、再アタッチで続きから、というモデル。

---

## 1. 「市長と直接チャット」の最短経路

**`gc session attach <id-or-alias>`**（`cmd/gc/cmd_session.go:839`）

```bash
gc session attach mayor    # alias 指定（"mayor" は config の named_session）
gc session attach gc-42    # session ID 指定
```

> *"Attach to a running session or resume a suspended one. If the session is active with a live tmux session, reattaches."* (cmd_session.go:840-845)

Gas City が tmux 内で起動した **Claude Code（または Codex）のインタラクティブ画面そのもの** にアタッチ。「市長役の Claude Code セッション」に直接入って、いつもの Claude Code として対話できる。

サスペンド済みなら、プロバイダの resume 機構（Claude: `--resume`, Codex: `resume` subcommand 等）を使って復帰、tmux セッション死亡時はリスタートまで自動でやる。

---

## 2. アタッチせずにメッセージだけ送る

| コマンド | 用途 | ソース |
|---------|------|--------|
| `gc session nudge mayor "<text>"` | ターミナルにタイプするのと等価のテキスト送信 | `cmd_session.go:1408` |
| `gc session submit mayor "<text>"` | セマンティック送信（intent: default / follow-up / interrupt-now） | `cmd_session.go:78` |
| `gc mail send mayor "<body>"` | 永続メール（beads 上に保存される非同期メッセージング） | `cmd_mail.go:700` |

`submit` には HTTP API もある（`internal/api/client.go:338` の `SubmitSession`）。外部ツールから飛ばすことも可能。

`gc session submit` の intent オプション（`session.SubmitIntent`）:

| intent | 意味 |
|--------|------|
| `default` | 通常送信（セッションがアイドルなら即送、busy なら待機） |
| `follow-up` | 直前の応答へのフォロー扱い |
| `interrupt-now` | 走行中でも割り込み送信 |

## 3. ライフサイクル管理

| コマンド | 用途 |
|---------|------|
| `gc session new <template>` | 新しい役割のセッションを on-demand で起動（例: 市長を 1 人立ち上げる） |
| `gc session list` | 走っているセッション一覧 |
| `gc session peek <id>` | 起こさずにステータスだけ覗く |
| `gc session suspend <id>` | アイドル時にサスペンド |
| `gc session close <id>` / `kill <id>` | 終了 |
| `gc session rename <id> <title>` | 表示名変更 |
| `gc session prune` | 整理 |

`[chat_sessions]` セクションの `idle_timeout`（`internal/config/config.go:1150-1156`）でデタッチ済みチャットセッションを自動 suspend する仕組みもある（`cmd/gc/chat_autosuspend.go`）。

```toml
# city.toml
[chat_sessions]
idle_timeout = "30m"   # 30 分アタッチがなければサスペンド
```

## 4. ダッシュボード（Web UI）— **読み取り中心**

`cmd/gc/dashboard/web/src/panels/crew.ts` を確認した結果:

- ✅ クルー一覧（mayor, deacon, coder などが表で並ぶ）、Attached / Detached バッジ表示
- ✅ 「📎 Attach」ボタン → クリックすると `gc agent attach <template>` というコマンドが**クリップボードにコピー**される（ブラウザ内ではアタッチしない）
- ✅ 「Log」ボタン → SSE で `/v0/city/{cityName}/session/{id}/transcript` をストリームし**過去のトランスクリプトを表示**
- ❌ **ブラウザから直接メッセージを送るチャット入力フォームは無い**
  - `grep` で `submit-session` / `composer` / `message-input` / `chat-input` どれも 0 件
  - ダッシュボードコードから `SubmitSession` 呼び出しも 0 件

ダッシュボードは「観測 + アタッチコマンドの copy 補助」までで、Claude Code のような**チャット UI そのものではない**。実対話はターミナルに戻って `gc session attach mayor` を叩く流れ。

ただし HTTP API（`SubmitSession`）は用意されているので、Web チャット UI を独自に実装することは可能。

## 「市長とチャットしたい」のニュアンス別まとめ

| ニュアンス | 可能か | 手段 |
|-----------|--------|------|
| 立ち上がってる市長セッションに入って Claude Code として対話 | ✅ YES | `gc session attach mayor` |
| ターミナルに入らず一発メッセージを送る | ✅ YES | `gc session nudge mayor "..."` / `submit` |
| 永続的な手紙を送る（後で読ませる） | ✅ YES | `gc mail send mayor "..."` |
| ブラウザ上でチャット UI として往復 | ⚠️ NO（観測とログ表示のみ。送信 API はあるがダッシュボードからは未使用） | (要自前実装 or `submit` API を別 UI から叩く) |
| 「市長」役自体を立ち上げる | ✅ YES | `gc session new mayor` |

## 設計上の含意

Gas City のチャット UX は以下の前提に立っている:

1. **Claude Code / Codex 自体が既に優秀な対話 UI である** → 重複実装しない（ZFC 思想）
2. **役割（mayor / deacon / coder 等）はセッションのラベル** → `attach mayor` で目的の Claude Code 画面に飛び込む
3. **永続性は tmux と beads で担保** → 切断してもセッションは生き続ける、メールは beads 上に永続
4. **観測は Web ダッシュボード、操作はターミナル** → 役割分担

ChatGPT 風の Web チャット UI が欲しい場合は、`SubmitSession` HTTP API + transcript SSE を使って自作するか、単に tmux + ターミナルの世界観を受け入れる必要がある。

## 次のアクション候補

- **手元で試すなら**: `gc start examples/swarm/` → `gc session list` → `gc session attach mayor` の順で対話開始。
- **Web チャット UI を作るなら**: `internal/api/client.go` の `SubmitSession` と `/v0/city/{cityName}/session/{id}/transcript` SSE を組み合わせた最小フロントを書けば実装可能。
- **非同期メッセージング中心の運用**: `gc mail send` + `gc mail inbox` で beads 上に永続化されたメールスレッドを構築する用途も視野。
